第 10 章

LLM 训练流程概述

🌰 先讲个直觉

想象一个小孩，你给他看一万本教科书，他学会了写文章——这叫预训练，他会写了但写的可能乱七八糟。然后你给他看一百篇范文让他模仿格式——这叫SFT，他学会了按"问题→回答"的格式写。最后你不给范文了，只是他每次写完之后你表扬或批评——这叫RLHF，他学会了写"让人喜欢"的内容。

预训练让模型会说话，RLHF 让它说得讨人喜欢。

三阶段训练流程

# LLM 训练三阶段
Stage 1: 预训练 (Pre-training)
  → 自回归预测下一个 token
  → 学习语言的通用统计规律
  → 输出：Base Model（会说话，但不太可控）

Stage 2: 监督微调 (SFT)
  → 给定指令，预测回答的下一个 token
  → 让模型学会遵循指令格式
  → 输出：SFT Model（会按格式回答）

Stage 3: RLHF (基于人类反馈的强化学习)
  → 用 RL 优化模型使其符合人类偏好
  → 三种方法：DPO / PPO / GRPO
  → 输出：Aligned Model（回答既正确又讨人喜欢）

SFT：为什么只训几轮？

SFT 也是"预测下一个词"，但只在回答部分计算损失（指令部分不参与）。

⚠️ 灾难性遗忘

SFT 一般不会训练很多轮！如果训练太多，模型会过拟合微调数据，丢失预训练阶段学到的通用能力——因为 SFT 目标函数中没有约束新旧模型的 KL 散度。

LLM 即 RL 环境

在 RLHF 中，LLM 就是强化学习的"智能体"：

# LLM = RL 环境中的智能体
策略 πθ    = LLM
状态 Sₜ    = 上下文（已生成的文本）
动作 Aₜ    = 输出一个 token
动作概率   = πθ(token | context)
奖励 Rₜ    = 奖励模型的评分 - KL散度惩罚
机器学习类比RLHF = 给 LLM 装上"判分老师"
SFT 监督微调给 LLM 看示范答案，模仿回答
↔
监督学习 / 微调有标注的标准答案
RLHF 强化学习不告诉答案，只打分，自己探索
↔
主动学习 / 弱监督没有精确标签，只有反馈信号
类比要点：SFT 像传统监督微调——有人工标注的标准答案。RLHF 像弱监督——人类评判"好不好"比写"标准答案"容易得多，所以能大规模收集反馈。这就是 OpenAI 所说的"如何用数学定义'乐于助人'"。

三种 RLHF 方法对比

方法一

DPO — 直接偏好优化

不需要奖励模型，直接从偏好数据学习。LLM 本身就是奖励模型。最简单，但效果有限。

 方法二

PPO — 复刻 InstructGPT

SFT → 训练奖励模型 → PPO 微调。OpenAI 的经典方案，最成熟但需要额外训练 Critic。

 方法三

GRPO — 复刻 DeepSeek-R1

不需要 Critic，用组相对优势。专为推理任务设计的 RL 方法，DeepSeek 的核心技术。

DPO — 直接偏好优化

🎯 导读

DPO 的核心洞察：你的大语言模型实际上就是一个奖励模型。不需要单独训练 RM，直接从偏好数据优化 LLM。就像拿两份作业比较——"A 比 B 好"比给 A 打 8 分更容易、更可靠。

机器学习类比DPO ↔ 对比学习 / 排序学习
DPO 偏好数据chosen 和 rejected 成对比较
↔
对比学习正负样本对anchor + positive vs negative
拉开 chosen/rejected 概率偏好回答概率 > 厌恶回答
↔
对比损失拉近/拉远嵌入正样本靠近，负样本远离
类比要点：DPO 和 CLIP、SimCLR 的思路惊人相似——都是通过"成对比较"学习。DPO 拉开的是回答概率，对比学习拉开的是嵌入距离。这种"相对比较"比"绝对打分"更容易收集数据。

偏好数据集

# 偏好数据格式
{
  "prompt": "这部电影怎么样？",
  "chosen": "这部电影很好看，推荐！",     # 人类偏好
  "rejected": "这部电影不好看。"     # 人类厌恶
}

DPO 目标函数

J_DPO(θ; πref) = E[ log σ( β · ( log[πθ(yw|x)/πθ(yl|x)] − log[πref(yw|x)/πref(yl|x)] ) ) ]

yw=偏好回答，yl=厌恶回答，πref=冻结的参考模型，β=超参数。当模型对偏好回答的概率相对厌恶回答上升时，目标函数增大。完整推导见DPO目标函数推导。

DPO 的局限性

容易过拟合：训练太多模型会崩溃（chosen→1, rejected→0），不能多轮。
偏好循环：数据中 A>B 和 B>A 同时存在时无法学习。
DPO 算 RL 吗？本书作者认为不算——它需要预定的偏好数据集，而真正的 RL 是策略自己采样数据。

PPO 微调 LLM — 复刻 InstructGPT

🎯 导读

OpenAI 的 InstructGPT（ChatGPT 的前身）使用经典的三步流程：SFT → 训练奖励模型 → PPO 微调。这套流程是 RLHF 的"黄金标准"。

# InstructGPT 三步流程
Step 1: SFT — 有监督微调
  输入: Prompt → 输出: 人工编写的回复
  → 学会基本的回答格式

Step 2: RM — 训练奖励模型
  SFT 生成 4-9 个回复 → 人工排序 → 训练 RM
  → 学习"什么是好回答"

Step 3: RL — PPO 强化学习
  LLM 生成回复 → RM 打分 → PPO 更新参数
  + KL散度惩罚（防止偏离SFT太远）
  → 让 LLM 朝着"人类喜欢"的方向进化
机器学习类比InstructGPT 三步 = 监督预训练 + 训练评估器 + 对抗微调
Step 1: SFT 监督微调学习基本的回答格式
↔
预训练 / 初始化给模型一个合理的起点
Step 2: RM 训练奖励模型学习"什么是好回答"
↔
训练评估指标 / 判别器定义"什么是好模型"
Step 3: PPO 强化学习优化 LLM 使 RM 打高分
↔
对抗训练 / GAN优化生成器骗过判别器
类比要点：Step 2+3 的组合本质上就是 GAN——训练判别器区分好坏，再训练生成器骗过判别器。区别：GAN 同时训练对抗，InstructGPT 先训好 RM 再用它指导 LLM，且加 KL 约束防止跑偏。

即时奖励 Rₜ 的精妙设计

奖励模型只能给完整序列打分，不能给单个 token 打分。怎么把"序列级奖励"转化为"token 级奖励"？InstructGPT 的做法：

非最后一个 token：Rₜ = −β · log[ πθ(yₜ|...) / πref(yₜ|...) ]
只有 KL 惩罚，没有 RM 分数

最后一个 token（<eos>）：
R_T = RM分数 − β · log[ πθ(y_T|...) / πref(y_T|...) ]
RM 分数在序列结束时一次性给出

RM 分数只在序列末尾出现（稀疏奖励），靠 GAE 把最后的奖励"抹匀"分给前面每个 token。

关键秘诀：KL 散度惩罚

RM 分数推模型往"人类喜欢"的方向走；KL 惩罚拽住模型不让它跑偏。没有 KL 约束 → 奖励黑客：模型生成乱码骗高分。InstructGPT 成功的关键就是两者的平衡。

GRPO 微调 LLM — 复刻 DeepSeek-R1

🎯 导读

DeepSeek 用 GRPO 在推理任务上取得了突破。本章使用 DAPO（GRPO 变种）微调 Qwen2.5-3B，从零复刻类似 DeepSeek-R1 的推理模型，让模型产生思维链。

核心理念

GRPO 训练推理模型像"自监督的多路投票"——对每道题生成多个解答，对的强化、错的削弱。模型逐渐学会"怎么思考才能答对"。注意：这和 GAN 不同，GRPO 的奖励来自客观正确性验证（数学题能验证答案），不需要人类标注。

机器学习类比GRPO 训练推理 = 自监督 + 多路采样投票
对同一问题采样 G 条回答生成多个候选，比较优劣
↔
多路采样 / Beam Search生成多个候选，选最优的
用正确性作为奖励答案对=正奖励，错=负奖励
↔
自监督学习用数据本身验证模型输出
类比要点：GRPO 训练推理像"自监督的多路投票"。DeepSeek-R1 靠此低成本获得强推理能力——不需要人类标注思维链，只需数学题的正确性作为奖励。

DAPO 算法流程

# DAPO 伪代码（GRPO 变种，字节提出）
for each training step:
    1. 随机选取 N 个问题: q₁, q₂, ..., qₙ
    2. 对每个问题 qᵢ，采样 G 条回答: aᵢ,₁, ..., aᵢ,G
    3. 计算每条回答的奖励 rᵢ,ⱼ（正确=1，错误=0等）
    4. 组内归一化: μᵢ = mean(rᵢ,₁,...,rᵢ,G)
                 σᵢ = std(rᵢ,₁,...,rᵢ,G)
    5. Token 级优势: Aᵢ,ⱼ[t] = (rᵢ,ⱼ - μᵢ) / σᵢ
    6. 使用 GRPO 目标函数计算梯度并更新

DAPO 的两项关键改进

  • Token 级策略梯度损失：每个 token 的优势 = 其所在轨迹的组相对优势
  • 移除 KL 散度：不需要参考策略网络，减少 GPU 内存

GRPO 应用场景

场景一

医疗思维链

使用 GRPO 训练 LLM 产生结构化的医疗推理过程，提升诊断准确性。

场景二

Text-To-SQL

用 GRPO 优化 LLM 的 SQL 生成能力，以执行正确性作为奖励信号。

为什么 GRPO 适合推理任务？

推理任务（数学、代码、逻辑）有明确的正确性标准，适合作为奖励信号。GRPO 的"多条采样取最优"天然适合——对同一问题生成多个回答，好的强化、差的削弱。

📌 一句话记住 RLHF：预训练让模型会说话，RLHF 让它说得讨人喜欢。三种方法——DPO 最简单（不需要 RM）、PPO 最成熟（需要 Critic）、GRPO 最省内存（用组优势替代 Critic）——DeepSeek-R1 的推理突破靠的就是 GRPO。
